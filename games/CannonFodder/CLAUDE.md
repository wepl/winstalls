# CannonFodder — game internals & reverse-engineering notes

Working notes on how the game works, for debugging the WHDLoad slave. Built up
while fixing relocation crashes (June 2026). **All offsets are exe-image offsets**
(= runtime address minus the load base). The disassembly `FODDERC.asm` is the
**en1** version; labels `lbX0NNNNN` encode the offset (e.g. `lbC0153B4` = offset
`$153B4`). Tested version: **en1, CRC `$b157`** (others: en2 `$a1c0`, de `$7b22`,
fr `$c3ce`, it `$5723`). Wepl normally tests only en1; finish en1 before the rest.

## Relocation architecture (the whole point of the slave)

- The game exe `fodderc` was originally hard-linked at `$80000`. The slave now
  relocates it to an arbitrary base (ExpMem). `EXELEN=$80000`, `PICLEN=$b000`.
- Disassembly `FODDERC.asm` is built for base `$80000` (`START=$80000`).
- `fodder.rel` = relocation table, generated by the **imager** (`cannonfodder.islave.asm`)
  by diffing two builds on disk: `FODDERC` (base `$80000`) vs `FODDERF` (base `$200000`).
  A base-relative long differs between the two builds by **exactly `$180000`**
  (high-word delta `$18`, low word unchanged). The imager's detector currently also
  gates on the fodderc high word being `$0008..$000F`; that window **misses**
  base-relative values ≥ base+`$80000` (high word `$0010`, e.g. `lea $100000,sp`)
  and the copylock-encrypted entropy blocks. The truly robust test is "long delta
  == `$180000`". These misses were dormant in the crashes investigated (those code
  paths weren't active), but keep it in mind.
- Slave relocation loop (`_start`): loads `fodder.rel`, then for each listed offset
  `add (base-$80000)` to the long there.
- `_team` = `expmem+$1556`; `_picmem` = `expmem+EXELEN`. BaseMemSize `$80000`.

## Crash-dump workflow

WHDLoad writes three files on a fault (and appends per fault):
- `.whdl_register` — registers + exception frame (068 access-fault frame: SR, PC,
  format/vector, fault address, FSLW). The "regular stack" shown is the **active**
  stack (game runs in **supervisor** mode, SR `$2000`, SP≈SSP; USP is unused-ish).
- `.whdl_memory` — memory from address 0 (BaseMem).
- `.whdl_expmem` — memory from the ExpMem base. **ExpMem file offset X = exe-image
  offset X**, so it maps 1:1 onto `FODDERC.asm` offsets.

## RNG (critical: explains relocation-sensitivity)

- `random2` (entry `$1D850`): walks a pointer (`random2_adr`) through the game's
  **own code**, reading it as entropy; bounded by `cmpa.l #$ACA14` (= base+`$2CA14`
  = end of exe), reset target `#lbC005D92` (`$5D92`). `random2_adr`, the bound and
  the reset target are all in `fodder.rel` and relocate correctly.
- `random1` (entry `$1D810`) has no callers; disabled via `PL_I $1d810`.
- **Because random2 reads relocated code, the RNG sequence necessarily changes when
  the exe is relocated** (the relocated address bytes in the code differ). This is
  expected/benign in itself, but it steers the game into **different states/paths**
  than the fixed-`$80000` original — which is how latent bugs get unmasked. So a
  relocation-only change altering behaviour is normal; don't chase speed/caches.

## Object system

- **Object array** at offset `$7DB2C` (BSS, above the exe). Entries are `$76` bytes,
  walked until `word0 == $FFFF` (terminator). `word0` positive = active object,
  negative (bit15) = skip, `$FFFF` = end.
- **Dispatcher `lbC0153B4`** (`$153B4`): `lea ($FDB2C).l,a0` then loop:
  `tst.w (a0)` → if active, `lea (lbL014608).l,a1; move.w ($18,a0),d1; asl.w #2,d1;
  movea.l (a1,d1.w),a1; jsr (a1)` then `lea ($76,a0),a0`. **Every handler MUST
  preserve A0** (the object pointer) — the loop relies on it. The `jsr (a1)` is at
  `$153D6`; the preceding `movea.l (a1,d1.w),a1` at `$153D2`.
- **Jump table `lbL014608`** (`$14608`): **111 entries** (index `0..110`),
  index = `field $18 × 4`. Out-of-range `field $18` → reads past the table → garbage
  pointer → bad jump. Entry [0]=`$16730`, entry [5]=`$1DADE`.
- Object fields seen: `+$18` = type (table index); `+$1A`,`+$6A`,`+$70` = pointers;
  `+$10`,`+$12`,`+$22`,`+$38`,`+$44`,`+$45`,`+$5D`,`+$5E` = state/counters; many
  coordinate words. Object **type 5** handler `$1DADE` → `lbC015BB2` (big per-object
  update) → … → `lbC016CAA` → `lbC016D7C` (team handling).

## Team / player data

- `actual_player_team` at offset `$1556` (word; `0..5`, or negative/`$FFFF` when
  invalid — several access-fault fixes guard exactly this).
- `lbL005C16` (`$5C16`): table of per-team struct pointers, team0=`$181C`,
  stride `$78` (`$181C,$1894,$190C,…`). Relocated.
- `team_table` (~`$5C16` family, label at the `dl lbL005B26` near offset `$5C16`):
  `team_table[team]` → pointer to a team struct; `lbL005B26` (`$5B26`) = team AI/order
  table.
- `lbL001818` (`$1818`): pointer = `lbL005C16[team]` (current player's team struct).
- Team waypoint/order lists are `$FFFF $FFFF`-terminated.

## Slave patches of note

- Copylock/SMC disabled: `PL_I/PL_R` at `$7a14 $7aaa $a6a8 $c3c6 $1fc92 $2c044`.
- Cold-start skipped: `PL_S $5d92` (start code sets `sp=$100000`, bangs hardware).
- `_keyboard` (`PL_P $a2f8`): keyboard IRQ reroute + trainer keys (`N`=skip level,
  `HELP`=grenades/bazookas, key `$46`=Del=exit).
- `_loader`/`_gettmp`/`_loadgame`/`_savegame`: disk/file & savegame emulation.
  Savegames store absolute pointers into the relocated exe → fixed up in `_loadgame`.
- **Access-fault fix handlers** (replace one instruction via `PL_PS`, must emulate it
  and return to offset+6; cannot overwrite a branch target):
  - `_af0` (`$16d7c`) and `_af1` (`$1eb36`): guard negative `actual_player_team`
    before the `team_table[team]` deref. They read the team via `(_team)`. **They must
    not clobber A0** (it is the object pointer when reached from the dispatcher) — they
    use **A1**, which the following original instruction (`lea team_table,a1`)
    overwrites anyway.
  - `_af3` (`$163c2`) / `_af4` (`$1a83a`): index/pointer bounds guards (currently
    commented out by Wepl).
  - `_s1` (`$243ee`): savegame pointer range fix.

## Fixed bug (27.06.2026) — reference case

Symptom: address error / illegal jump after relocating the exe off `$80000`.
Root cause: `_af0`/`_af1` (Wepl's own access-fault fixes) did `move.l (_team),a0` and
never restored A0. At `$16d7c` (reached via object-type-5 dispatch) A0 is the object
pointer the dispatcher `lbC0153B4` requires. So A0 became `(_team)` (`$175556` in the
dump) → next loop iteration read a garbage `field $18` (`$8E`=142 > 110) → jumped past
`lbL014608` to `$140004`. Only surfaced after relocation because the changed RNG path
reached `$16d7c` in object-type-5 processing. **Not** a `fodder.rel` error (all real
pointers + RNG verified correctly relocated). Fix: use A1 instead of A0 in `_af0`/`_af1`.

Diagnostic technique that found it: wrap the dispatcher's `jsr (a1)` (patch at
`$153D2`, returns to `$153D8`) to save A0, call the handler, and `illegal` if A0
changed — leaving the culprit handler's offset in D1. (Kept commented out in the
slave as `_diag` for reuse.)
